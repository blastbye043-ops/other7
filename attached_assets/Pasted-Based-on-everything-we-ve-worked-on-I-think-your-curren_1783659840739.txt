## I also think I know the underlying cause

From all the errors you've shared over the last few days, I don't think the main problem is the frontend. I think the backend architecture is causing the delays.

Your backend appears to:

* Launch a new `yt-dlp` process for every request.
* Wait for the full output before responding.
* Not cache metadata.
* Return a generic 500 error instead of the actual cause.
* Possibly perform database work before sending the response.

Those design choices can make the site feel slow and fragile.

If you want the performance of sites like **SaveFrom**, **YTDown**, or **SnapScooper**, the backend should be restructured around:

* Fast metadata extraction (`yt-dlp --dump-single-json` or equivalent).
* Short-lived metadata caching by video ID.
* Non-blocking database operations (or none at all for metadata).
* Immediate response with basic info, followed by loading formats if needed.

That combination will give a much faster and more reliable user experience.
